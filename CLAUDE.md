# Beauty Tech Japan プロジェクト固有ルール

## 画像パフォーマンス（自律対応）
- `unoptimized: true` は絶対に設定しない。Vercelの画像最適化を常に有効にする
- `next/image` の `sizes` プロップは実際の表示サイズに合わせて正確に設定する
- 外部画像ホスト（Unsplash等）は `remotePatterns` に必ず登録する
- `formats: ["image/avif", "image/webp"]` を常に有効にする
- ファーストビューの画像には `priority` を付ける
- 作業の中で画像パフォーマンス問題を発見したら、ユーザーに言われる前に自分で修正する

## UI/UX（自律対応）
- **PCとスマホで臨機応変にレイアウトを整える**（ユーザーに言われなくても自分で判断して対応する）
- スマホ（375px）・PC（1280px）両方で見た目を常に意識する
- PC では情報を横並びで広く使い、スマホでは縦積み・大きいタップ領域で見やすくする
- 横スクロールが発生しないよう `overflow-hidden` や `flex-wrap` を適切に使う
- フォーム・ボタンはスマホで切れないよう `flex-col sm:flex-row` パターンを使う
- カテゴリフィルターなど多数の要素は `flex-wrap` で折り返しにする
- グリッドはスマホで `grid-cols-1` または `grid-cols-2`、PCで `grid-cols-3` 以上に調整する
- テキストサイズはスマホで読みやすく（最小 `text-xs`）、PCで適切に大きくする
- 作業の中でUI/UXの問題を発見したら、ユーザーに言われる前に自分で修正してデプロイする
- フロントエンドを変更したらデプロイ前に Playwright でスクリーンショットを確認する

## 自律ブラッシュアップ
- 作業の合間にサイト全体を見渡し、改善できる点があれば自分で実施する
- 特に以下を定期的にチェックして改善する：
  - 画像の読み込み速度・最適化設定
  - モバイルレイアウトの崩れ
  - 誤ったラベル・テキスト（例：「人気順」が実は「古い順」など）
  - リンク切れ・未実装の機能（ボタンはあるがページがないなど）
  - About/説明ページの内容が実態と合っているか

## デプロイ
- 作業完了後は必ず `vercel --prod` でデプロイする（git push だけでなく）
- デプロイ後は `Invoke-WebRequest https://beauty-tech-japan.vercel.app/ -UseBasicParsing` でHTTP 200を確認する
- 200以外が返ったらすぐ原因を調査して修正する

## セッション開始時の自動チェック（毎回実行）
- `scripts/data/posted_ids.log` の行数と `scripts/data/beauty_topics.json` の件数を比較し、残りトピック数を確認する
- 残り30件以下なら自動的に新トピックを100件生成して追加する
- 最終記事の `publishedAt` を確認し、7日以上更新がなければ GitHub Actions の状態を確認する
- Vercel の最新デプロイが正常かを確認する

## API予算の監視（Gemini API）
- **$10を超えたら即座にユーザーに報告する**（記事生成も自動停止される）
- **$8を超えたら事前に警告する**
- セッション開始時に `scripts/data/api_cost_log.json` を確認して累計コストをチェックする
- 累計コストが $8 以上なら会話の最初にユーザーへ報告する
- コスト確認コマンド: `python -c "import json; log=json.load(open('scripts/data/api_cost_log.json')); print(f'累計: \${log[\"total_usd\"]:.3f}')"`
- 実際の請求は Google Cloud Console / Gemini API ダッシュボードで確認する

## トピック残量の監視・補充
- セッション開始時に必ず残量を確認する
- 残り30件以下：即座に Gemini API で100件生成して `beauty_topics.json` に追加する
- 残り10件以下：緊急アラートとして先にユーザーに報告する
- トピック補充後は必ずコミット＆デプロイする

## SEO・MEO・パフォーマンス（高いパフォーマンスを常に発揮する）
- **SEO**: 新ページには必ず `title`・`description`・`openGraph`・`twitter` の4点セットを設定する
- **SEO**: 構造化データ（JSON-LD）を記事ページ・トップページに正確に設定する
- **SEO**: `<h1>` は1ページに1つ、`<h2>/<h3>` で情報を階層化する
- **SEO**: canonical URL・robots meta を適切に設定する
- **SEO**: 内部リンクを意識的に張る（関連記事・タグページへのリンク）
- **パフォーマンス**: Core Web Vitals（LCP・FID・CLS）を常に意識する
- **パフォーマンス**: ファーストビューのLCPを改善するため `priority` 画像を正しく設定する
- **パフォーマンス**: 不要なJavaScriptを減らし、Clientコンポーネントは必要最小限にする
- 上記を作業の中で問題を発見したら自律的に修正する

## 記事品質の自律チェック
- 生成記事は本文400字未満・タイトル空・要約空の場合は品質不足と判定して再生成する
- `post_one_article.py` の品質チェック機能を活用する
- 品質不足が連続した場合はプロンプトを見直して改善する

## TypeScript品質
- `strict: true` は常に維持する（`tsconfig.json` で設定済み）
- `any` 型は使わない。型が不明な場合は `unknown` を使って型ガードを書く
- 新しいコンポーネントにはPropsの型定義を必ず書く
